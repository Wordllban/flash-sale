# 03 — Redis module + atomic Lua reserve script

**Phase:** 1 · **Depends on:** #01 · **Implements:** ADR-001, ADR-002, ADR-004

## Goal
Create the Redis connection module and the single **atomic Lua script** that is the heart of the
engine: check stock + check not-already-bought + decrement + record buyer + append outbox intent —
all indivisibly.

## Concepts you'll learn
- **ioredis** connection + `defineCommand` to register a reusable Lua script.
- **Why Lua is atomic** in Redis: a script runs start-to-finish with nothing interleaved.
- **Redis Streams** (`XADD`) used as a durable outbox inside the same atomic op (ADR-002).
- Keys vs args (`KEYS[]` / `ARGV[]`) and return-code conventions.

## Steps
1. `pnpm add ioredis`. Create `RedisModule` providing a configured `ioredis` client (from `REDIS_URL`),
   with shutdown handling. Wire the real Redis indicator into `/health`.
2. Add the reserve script `src/redis/scripts/reserve.lua`:
   ```lua
   -- KEYS[1]=stock:{saleId}  KEYS[2]=buyers:{saleId}  KEYS[3]=orders:stream
   -- ARGV[1]=userId  ARGV[2]=saleId  ARGV[3]=orderId
   if redis.call('EXISTS', KEYS[1]) == 0 then return -2 end          -- sale not active
   if redis.call('SISMEMBER', KEYS[2], ARGV[1]) == 1 then return -3 end -- already bought
   local stock = tonumber(redis.call('GET', KEYS[1]))
   if stock <= 0 then return -1 end                                   -- sold out
   redis.call('DECR', KEYS[1])
   redis.call('SADD', KEYS[2], ARGV[1])
   redis.call('XADD', KEYS[3], '*', 'orderId', ARGV[3], 'userId', ARGV[1], 'saleId', ARGV[2])
   return 1                                                           -- success
   ```
3. Register it via `redis.defineCommand('reserve', { numberOfKeys: 3, lua })` and expose a typed
   `InventoryService.reserve(saleId, userId, orderId)` returning a result enum
   (`OK | SOLD_OUT | ALREADY_BOUGHT | NOT_ACTIVE`).
4. **Note:** the `orderId` is generated by the caller (controller, #07) and passed in, so it is
   stable across stream redeliveries — the consumer's idempotency key.

## Acceptance criteria
- [ ] `RedisModule` connects, appears in `/health`, disconnects on shutdown.
- [ ] `reserve` returns 1 and decrements once for a fresh user with stock.
- [ ] Second call by the same user returns `ALREADY_BOUGHT` and does **not** decrement.
- [ ] Calls beyond stock return `SOLD_OUT`; stock never goes negative.
- [ ] On success, an entry exists in `orders:stream` (`XLEN` increases).
- [ ] Unit test fires N concurrent reserves at stock=5 → exactly 5 successes.
